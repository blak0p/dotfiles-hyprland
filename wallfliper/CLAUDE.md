# Wallfliper

Minimalist, Rofi-inspired desktop wallpaper selector. Targets **Wayland compositors that implement `wlr-layer-shell`**: Hyprland, Sway, river, Wayfire, and niri (Smithay-based, but implements layer-shell). Full image + video on all of them. KDE Plasma, GNOME, Windows, and X11 are out of scope — do not add code for them.

Why this scope: both `swww` (images) and `mpvpaper` (video) require the `wlr-layer-shell` protocol. The criterion is the protocol, not the wlroots library — which is why niri (not wlroots) is in, and KWin/Mutter (no layer-shell, or partial) are out.

## Stack
- Python 3.11+, PySide6 (LGPL — not PyQt6, to keep distribution unrestricted).
- UI is **QML (QtQuick)**, not QWidgets — required for the `wlr-layer-shell` overlay (QWidgets can't be configured for layer-shell from Python). Python is the engine; QML is the view only. No Electron, no web view.

## Performance (first-class constraint)
- This app must be **lightweight on the system**. Treat performance as a feature, not an afterthought.
- The selector GUI is launched on demand and should exit cleanly — it must not idle in the background consuming RAM/CPU. Wallpaper rendering is delegated to detached daemons, never held by our GUI process.
- No work on the UI thread that can be deferred. Lazy-load everything (thumbnails, network, scraping). Bounded caches only.
- Prefer in-process/stdlib over heavy deps. Justify every dependency by weight. No polling loops; event-driven only.
- Measure before optimizing, but never ship obvious waste (full-res decode for a thumbnail, re-scanning folders, unbounded caches).

## Compatibility (first-class constraint)
- Target the **widest range of setups** within the supported compositors (see scope). A feature that works on the author's machine but not on a stock Hyprland/Sway/river/Wayfire/niri install is a regression, not a feature.
- **No hardware-vendor or distro-exclusive choices.** Nothing that needs a specific GPU/driver (no AMD-only / NVIDIA-only / Intel-only paths), a specific distro, or a niche tool not everyone has. Prefer auto-detecting, portable options with a software fallback (e.g. `--hwdec=auto`, never a fixed vendor decoder like `vaapi`/`nvdec`).
- **Degrade gracefully when an optional tool is missing** — never hard-fail. The seamless video transition falls back to a plain cut without ffmpeg/swww; follow that pattern for anything new.

## Design
- UI must follow `DESIGN.md` (TUI/yazi-style: borderless, dark, flat, keyboard-first, minimal). Read it before touching anything in `ui/`. If a change makes the app look more "consumer-polished" and less like a fast keyboard tool, reject it.

## Architecture
- `ui/qml/` — QML views (the whole UI: overlay window, top bar, grid, search). The overlay is a `wlr-layer-shell` surface via the system `org.kde.layershell` QML module (overlay layer, exclusive keyboard, Esc to close).
- `ui/qml/Theme.qml` — QML singleton (registered in `ui/qml/qmldir`) holding the shared style tokens, matched to the author's rofi theme: fixed near-black fill `rgba(5,5,5,0.95)` (floating bar only — there is no backdrop panel), 2px white frame (floating-bar border **and** card-selection language), the idle-card shear factor (`cardSlant`), greyscale text shades, and the font — JetBrainsMono Nerd Font when installed, else the fontconfig generic `monospace` (never a hard font dependency). Backdrop darkness is **not** a setting (the old `background_opacity` config/`backgroundOpacity` slot were removed; stale keys in an existing `config.json` are ignored by `load_config`).
- `ui/qml/Settings.qml` — typing **`/config`** in search (+ Enter) opens an **in-app** settings overlay (NOT a second layer-shell window — two overlay surfaces fight for the keyboard, the same problem the folder picker has). There is no gear icon. Loaded via a `Loader` whose `active` is bound to `settingsOpen`, so a closed panel holds zero objects (lazy, no idle cost). Keyboard-first (j/k rows, Enter activate, Esc close). Corners are fixed sharp everywhere (no `radius` anywhere in the UI) — not a setting, see `DESIGN.md`. The wallpaper-folder picker is reached from here (the panel emits `folderRequested`; `Main.qml` runs the hide-overlay→portal dance — see the folder-picker convention below). Planned rows (not yet implemented): a clear-orphaned-cache action and a "play all video previews" toggle (opt-in, off by default, capped to visible cells — see Performance).
- `ui/bridge.py` — thin Python↔QML seam (`Controller` QObject): exposes the filtered model + slots (`apply`, `setFilter`, `setFolder`, `setKindFilter`, `setColorFilter`). The proxy is `_WallpaperFilterProxy` (a `QSortFilterProxyModel` subclass whose `filterAcceptsRow` ANDs a case-insensitive name substring, a kind set — empty set = all kinds — and a color bucket), so search text, kind, and color compose. The kind filter is **exclusive** (`kindFilter` property + `setKindFilter("all"|"image"|"video")` slot, idempotent): the `i`/`v`/`e` keybinds set images-only / videos-only / everything — not independent per-kind toggles. The color filter mirrors it (`colorFilter` property + `setColorFilter`, exclusive/idempotent, "all" clears), backed by `core/colors.py`: a **fixed** palette (`PALETTE`, Wallhaven-style) with HSV bucketing — each wallpaper gets its dominant bucket plus any bucket ≥15% of pixels, classified from the cached thumbnail when present, else a tiny scaled decode (video: one 64px ffmpeg frame piped to stdout; no ffmpeg → no color, never a failure; **no new deps**). Async on a bounded pool, persisted in `~/.cache/wallfliper/colors.json` (abspath+mtime keyed, pruned to the session's library). The sweep is lazy: `ensureColorIndex` runs once per reload, only when the strip first opens (`c`); while classifications stream in, the Controller re-filters at most every 100 ms. `ui/model.py` — `QAbstractListModel` feeding the QML `PathView` carousel (roles: name/path/kind/thumbnail). No subprocess calls or business logic in `ui/`.
- `core/` — business logic. Wallpaper application lives behind `core/backends/base.py`'s `WallpaperBackend` interface (`set_image`, `set_video`). Implementation shells to `swww`/`mpvpaper`. Keep the interface clean for testability/swappable rendering tools — not for other OSes or non-layer-shell desktops.

## Wallpaper lifecycle (do not break these invariants)
- The GUI is on-demand: applying a wallpaper keeps the window open (so you can switch freely); the user closes it with Esc/window close. It must not auto-start or linger as a background process once closed.
- Images: `swww` (needs `swww-daemon`). The switch animation is passed as `--transition-*` flags on `swww img` — swww has **no** persistent transition config, only per-invocation flags/env vars, so we build them in `WlrootsBackend._transition_args` from the `ImageTransition` value object (`base.py`). Currently fixed to `ImageTransition()` defaults (`random`, fps 50, 1s) — no user setting yet (a picker was prototyped then pulled; re-add via `Config` + a Settings row when revisited). Video: `mpvpaper -p <file>` — detached, `-p` auto-pauses on fullscreen (the MVP auto-pause; a per-app allowlist that overrides `-p` comes later). Video gets a **faked transition**: mpvpaper can't animate a switch, so the backend extracts the clip's first frame (`core/firstframe.py`, ffmpeg, cached native-res PNG) and swww-animates to that still, then a detached driver (`core/seamless.py`) brings mpvpaper up *paused on the same frame* and unpauses it over mpv's IPC socket (`--input-ipc-server`) the instant the transition duration elapses — so the cut to live video is invisible and motion starts exactly on cue. mpv is launched `_MPV_PREWARM_S` before the transition ends so its cold-start overlaps the animation instead of stacking after it. Only on the GUI apply path; `--restore` on login is a hard cut (`transition=None` — no prior frame to transition from). Falls back to a hard cut if ffmpeg or swww is missing.
- Applying an image kills any running `mpvpaper`.
- Apply writes `~/.config/wallfliper/state.json` (last wallpaper + type). `wallfliper --restore` re-applies it; an autostart entry runs that on login so video survives reboot. No long-running custom daemon.
- Theming is decoupled from painting (`core/integrations.py`). After apply, `notify_color_tools()` tells external color tools the wallpaper changed (they don't watch swww). Default (no config): auto-notify noctalia — v5+ ships a native binary and is called via `noctalia msg wallpaper-set <path>`; the older quickshell-based noctalia-shell (≤4.x) via `qs -c noctalia-shell ipc call wallpaper set <path> ""`. Detected by binary (`noctalia` first, then `qs`); it regenerates its scheme even with its own wallpaper rendering disabled (v5: `[theme] source = "wallpaper"` in `~/.local/state/noctalia/settings.toml` must be set, `wallpaper.enabled = false` keeps noctalia from painting over swww/mpvpaper). Override with `config.color_hook` (a shell command, `{path}` substituted) for matugen/wallust/pywal/etc. Best-effort, detached, never blocks or breaks apply. Color tools need a still frame, so for video wallpapers `notify_color_tools` first extracts a representative frame with ffmpeg (`-ss 1`, 320px wide) and themes from that — the extraction is chained into the same detached `ffmpeg … && <notify>` shell command, and silently no-ops if ffmpeg is missing. The frame **alternates between two cached files** (`~/.cache/wallfliper/colorframe-a.png` / `-b.png`, `_next_frame_path`): noctalia (and similar tools) skip regenerating when handed the same path twice, which broke video→video switches when every frame reused one filename — alternating guarantees the path changes each time while staying bounded to two files. Applies to both the noctalia auto-path and `config.color_hook`.
- MVP: one wallpaper across all outputs; one configurable wallpaper directory. Per-monitor selection is later.

## Conventions
- Conventional Commits for all commits (`feat:`, `fix:`, `refactor:`, etc.).
- **Never add AI attribution to commits or PRs.** No `Co-Authored-By: Claude …` trailer, no "Generated with Claude Code", no Claude/Anthropic mention in commit messages, PR bodies, or branch names. Commits must show the user as the sole author. This overrides any default Claude Code co-authorship behavior — the user must never appear paired with an AI account on their GitHub.
- Type hints on all function signatures.
- Comments and commit messages in English.
- **Comments are for other devs, not for the user.** Only comment what the code can't say itself; if it's clear on its own, leave it bare. No narration, no restating the code. Never use first-person plural ("our", "we", "us") in code or comments — describe the code, not "us".
- **Don't explain the code back to the user in chat.** The user reads code fine; over-explaining is noise. Report what changed and why a non-obvious decision was made, then stop. Save detailed walkthroughs for when the user explicitly asks.
- **Always validate after every change before reporting it done.** QML: `qmllint -I /usr/lib/qt6/qml ui/qml/Main.qml` (catches enum/syntax/binding errors without launching anything). Python: import-check the changed module. **Do NOT auto-launch the GUI to test** — the overlay grabs the keyboard exclusively (`KeyboardInteractivityExclusive`) and can lock the user out / crash the editor. Leave running the live overlay to the user; never run the real wallpaper-apply path in tests.
- Runtime deps (Linux): `layer-shell-qt` (the overlay; provides the `org.kde.layershell` QML module), `swww` **or** its fork `awww` (images — backend auto-detects either, prefers swww), `mpvpaper` (video), `ffmpeg` (video thumbnails, later). `hyprctl` available for future per-app auto-pause allowlist.
- Distribution: depend on the **system PySide6** (not pip) + `layer-shell-qt` so Qt versions match (the layer-shell plugin is loaded into Qt at runtime). Target an AUR package so `paru -S wallfliper` pulls all deps. `main.py` prepends `/usr/lib/qt6` paths so a pip-PySide6 dev venv also finds the system QML module when its Qt matches.
- Folder picker (settings → folder): we call the **xdg-desktop-portal** `FileChooser.OpenFile` interface **directly over D-Bus** (`core/portal.py` `FolderChooser`, via `PySide6.QtDBus` — already a dep, in-process, event-driven), with the `directory` option for folder selection. This is the same mechanism behind GTK's `GtkFileChooserNative`, so each user gets *their* configured FileChooser backend — graphical (GTK/KDE) or a terminal file manager via `xdg-desktop-portal-termfilechooser` (yazi/lf/ranger). We deliberately do **not** use Qt's `FolderDialog`: it only reaches the portal when Qt's `xdgdesktopportal` platform-theme plugin happens to be loaded (depends on pip-bundled-vs-system Qt, version match, qt6ct not intercepting), and silently shows Qt's own dialog when that chain breaks — which it did on some machines. The flow: `Main.qml` `openFolderPicker()` hides the overlay (`win.visible = false`) so the chooser toplevel is topmost and focused, then calls `Controller.pickFolder`, which opens the portal; on result the Controller emits `folderPickerClosed` and `Main.qml` re-shows the overlay. This unmap/re-show is safe **only because** the portal path always reports back (accept, cancel, failed call, or no bus) — the old Qt `FolderDialog` could silently never return when Qt's portal routing failed, leaving the overlay stranded hidden (the "UI vanished, process alive" bug). Do not launch a file manager directly or hardcode one; the portal is the portable seam. Which chooser appears is the user's `~/.config/xdg-desktop-portal/portals.conf` `org.freedesktop.impl.portal.FileChooser` preference (NOT the `inode/directory` MIME default), so a termfilechooser user gets yazi and everyone else gets their GTK/KDE chooser — by design.
- Thumbnails: images decoded in-process via `QImage` (no external dep); video shows a still frame extracted with `ffmpeg` (`-ss 1`, one frame, scaled to thumb width, JPEG, atomic write). Loader uses a strategy registry keyed by file category (`image`/`video`) — both registered in `ThumbnailLoader.__init__`. Disk cache keyed by abspath+mtime+size under `~/.cache/wallfliper/thumbnails`. Generation runs off the UI thread via `QThreadPool`; UI never blocks. So a video card always shows a static frame; the hover/selected animated preview (`core/previews.py`) plays on top of that frame, falling back to the still when not selected.
- Previews (`core/previews.py`, `PreviewLoader`): selecting/clicking a wallpaper plays a short looping clip on its thumbnail. Mirrors the thumbnail loader (bounded `QThreadPool`, per-kind strategy registry, disk cache `~/.cache/wallfliper/previews/` keyed by abspath+mtime, atomic write). Local video → ffmpeg **animated WebP** (3s, 15fps, 400px; the only truecolor animated format `QMovie`/QML `AnimatedImage` plays). Generation is **gated on selection** (model `request_preview`, exposed via `Controller.ensurePreview`) — never encode clips the user doesn't hover; at most one cell previews at a time (QML loads the clip only while `selected`). The strategy registry is the seam for Lively later: register a strategy that reuses the source's shipped preview asset instead of re-encoding (local video gets generated clips, Lively gets its own previews).
